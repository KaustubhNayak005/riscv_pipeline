# Session Log: Phase 7 Implementation
**Date:** 2026-06-14
**Goal:** Implement Phase 7 (Run Small C Programs)

## Work Performed
- **RISC-V Toolchain:** Downloaded and installed the xPack GNU RISC-V Embedded GCC for Windows to `C:\tools\riscv-gcc\`.
- **Linker Script (`sw/linker.ld`):** Created a bare-metal linker script specifying a 4KB `ROM` (instruction memory) and 4KB `RAM` (data memory). Since the CPU is a strict Harvard architecture without a unified bus, the CPU cannot easily copy `.data` or `.rodata` from ROM to RAM. Therefore, initialized global data and read-only data are avoided in C.
- **Startup Assembly (`sw/crt0.S`):** Implemented boot code to initialize the stack pointer (`0x00001000`) and clear the `.bss` section before jumping to C `main()`.
- **UART Library (`sw/lib/uart.c`):** Created polling-based `putchar()` function mapping to the memory-mapped UART peripheral at `0x80000000`.
- **C Program (`sw/demos/hello_world.c`):** Implemented a simple "Hello World" application. Because of the Harvard architecture constraints, the string is built on the stack (`char msg[] = {'H', ...}`) to avoid using the `.rodata` section.
- **Build System (`sw/Makefile`):** Wrote a Makefile to compile the C source into `.elf`, copy it to a raw `.bin`, and format it into a `.mem` file (using a new python script `sw/bin_to_mem.py`) compatible with the Vivado `$readmemh` system.
- **Verification:** User verified the successful compilation and simulated the generated `sw/hello_world.mem` in Vivado using `tb_phase6`.

## Issue 3: Pipeline Hazard Causing Infinite Loop
- **Symptom:** The CPU executed the C program's startup code but infinitely looped back to `_start` (PC=0) immediately upon hitting `jal main`. The UART output was completely empty or repeated characters infinitely.
- **Root Cause:** A classic pipeline hazard in `top.sv`. When `jal` evaluated in the EX stage, it set `pc_sel=1` to branch. However, the instruction physically behind it in the pipeline (which was `ecall` located just past `main` in the binary) was in the ID stage. The ID stage correctly decoded `ecall` and asserted `trap_taken=1` combinationally. Because `top.sv` prioritized `trap_taken` over `branch_target`, the CPU took the trap (to `mtvec` = 0) instead of the branch, effectively throwing away the `jal` and restarting the program!
- **Fix:** Gated the ID-stage trap evaluation with `!pc_sel` and `!mret_exec`. A trap is now only valid if an older instruction is not currently flushing the pipeline.
- **Result:** Pipeline hazard eliminated. Branches correctly cancel younger exception triggers.
